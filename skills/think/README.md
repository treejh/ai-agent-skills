# think

새 기능, 설계, 아키텍처 결정 전에 호출하는 스킬. 거친 아이디어를 승인된 계획으로 전환하고 구조를 검증한다. 사용자가 승인하기 전까지는 코드를 작성하지 않는다.

> 출처: [obra/superpowers](https://github.com/obra/superpowers) (v3.8.0)

## 핵심 원칙

코드 작성 전에 계획을 세운다. 사용자가 승인할 때까지 코드, 스캐폴딩, 의사 코드를 작성하지 않는다.

의견을 직접 말한다. 입장을 취하고 어떤 증거가 생각을 바꿀지 명시한다. "흥미롭네요", "여러 방법을 생각해볼 수 있는데", "고려해볼 수 있습니다" 같은 표현은 피한다.

## 기능

- 2-3개의 접근 방식 + 트레이드오프 + 추천 제시
- 항상 최소(minimal) 옵션 포함
- 추천안에 대해 4가지 공격 각도로 검증:
  - **의존성 실패**: 외부 API/서비스 다운 시 graceful degrade 가능한가?
  - **스케일 폭발**: 데이터/사용자 10배 시 어느 단계가 먼저 깨지나?
  - **롤백 비용**: 방향이 틀렸을 때 되돌릴 수 있나?
  - **전제 붕괴**: 가장 취약한 가정은 무엇인가?
- 공격에 실패한 접근 방식은 공개적으로 폐기하고 이유 설명
- 승인 전 플레이스홀더(TBD, TODO) 없는 구체적 계획 요구

## 검증 체크리스트

- 8개 이상 파일 또는 새 서비스 추가 시 명시적으로 인정
- 3개 이상 컴포넌트 간 데이터 교환 시 ASCII 다이어그램 작성 (순환 탐지)
- 테스트 경로 전체 나열 (정상, 오류, 엣지 케이스)
- 데이터 변경 없이 롤백 가능한가?
- 필요한 API 키, 토큰, 서드파티 계정 사전 나열

## 사용법

```
/think
```

아이디어를 설명하면 접근 방식 2-3개와 추천안을 제시한다.

## 출력 예시

```
Approved design summary:
- Building: 사용자 세션 토큰을 Redis에 저장하는 인증 시스템
- Not building: OAuth 소셜 로그인, MFA
- Approach: Option B — JWT + Redis 블랙리스트 (롤백 가능, 스케일 안정)
- Key decisions:
  1. 토큰 만료 15분 / 리프레시 7일
  2. Redis 클러스터 대신 단일 인스턴스 (트래픽 현재 수준에서 충분)
  3. 블랙리스트 TTL = 리프레시 토큰 만료와 동일하게
- Unknowns: 스테이징 Redis 인스턴스 접근 권한 (인프라팀에 확인 필요)
```

## 언제 사용하나

- 새 기능 설계 전
- 아키텍처 결정을 내려야 할 때
- 복잡한 시스템 변경 계획을 세울 때

> **사용 안 할 때**: 버그 수정, 소규모 편집 (버그 디버깅은 `/hunt` 사용)
